# SENTINEL — Custom Intelligence Implementation Guide
## Split Between Two Parallel Agents

---

## The Big Idea

SENTINEL is not a security camera. It's a **programmable perception engine**. You give it a plain English instruction, and it rewires itself — what it looks for, what it stores, what it ignores, when it alerts you, and how it talks to you.

Examples of what a user types:

- "Count how many people enter and exit this hallway every hour"
- "Watch my grandfather — if he falls or doesn't move for 10 minutes, text me with a picture"
- "Count red backpacks passing through this area"
- "Monitor this intersection and tell me which direction gets the most foot traffic"
- "Watch this shelf — alert me when an item is removed"
- "Sit on my desk and just chat with me about whatever I'm working on"

That instruction becomes SENTINEL's entire personality, perception filter, storage schema, and alert system. Everything adapts.

---

## How the Custom Instruction Changes Everything

When a user gives SENTINEL an instruction, it changes four things:

### 1. What Gemini looks for (perception filter)
Default Gemini prompt: "Identify all people and objects in this frame."
Custom instruction "count red backpacks": Gemini prompt becomes "Focus on identifying backpacks. For each backpack, report its color. Flag any red backpacks with high confidence bounding boxes."

The Gemini vision prompt is dynamically rewritten based on the user's instruction. This means SENTINEL isn't wasting tokens analyzing irrelevant details.

### 2. What gets stored in MongoDB (smart schema)
Default: stores people count, bounding boxes, environment.
Custom instruction "count red backpacks": stores `{ timestamp, red_backpack_count, total_backpacks, locations, entering_or_exiting }`.

The database schema adapts to what the user cares about. No junk data.

### 3. When SENTINEL speaks or alerts (trigger conditions)
Default: speaks on every detection in scan mode.
Custom instruction "alert me if grandpa falls": ONLY speaks/texts when a fall is detected or no movement for 10 minutes. Silent the rest of the time.

### 4. What Featherless analyzes (pattern focus)
Default: general behavioral analysis.
Custom instruction "count foot traffic": Featherless prompt becomes "Analyze entry/exit patterns over time. Identify peak hours, trends, and anomalies in foot traffic."

---

## Modes (Simplified)

### Chat Mode
Talk to SENTINEL like a person. It watches but stays quiet unless you ask it something. "What do you see right now?" "How many people walked by in the last hour?" Good for desk companion, brainstorming buddy, or casual monitoring.

### Mission Mode
You've given SENTINEL a job. It executes silently and only alerts you when YOUR conditions are met. This is the airport counter, the grandpa monitor, the backpack tracker. It will text you, speak to you, or stay silent — whatever the mission requires.

### Scan Mode
Full autonomous demo mode. Gimbal sweeps, narrates everything, flags anomalies. For the hackathon judges.

### File 1: `lib/mission.ts` — The Mission Engine

This is the core of the custom instruction system. When a user types "count red backpacks," this file:
1. Sends the instruction to Gemini to parse into a structured mission
2. Generates a custom Gemini Vision prompt (what to look for in each frame)
3. Generates a custom storage schema (what fields to save in MongoDB)
4. Generates trigger conditions (when to alert the user)
5. Generates a custom Featherless prompt (what patterns to analyze)

```
State it manages:
  currentMission: {
    id: string,
    instruction: string,           // Raw user text
    mode: 'chat' | 'mission' | 'scan',
    
    // All of these are generated by Gemini from the instruction:
    visionPrompt: string,          // Custom Gemini Vision prompt
    storageSchema: object,         // What fields to extract and store
    triggerConditions: object[],   // When to alert
    featherlessPrompt: string,    // Custom pattern analysis focus
    speakBehavior: {
      onDetection: boolean,        // Speak on every detection?
      onTrigger: boolean,          // Speak when condition met?
      onPattern: boolean,          // Speak on pattern analysis?
      silent: boolean,             // Completely silent until asked?
    },
    alertBehavior: {
      textPhone: string | null,    // Phone number for SMS alerts
      email: string | null,        // Email for alerts
      includeFrame: boolean,       // Send a picture with the alert?
    },
    summary: {
      interval: number,            // How often to summarize (ms)
      lastSummary: Date,
    }
  }
```

  "storageDescription": "What each field means and what to extract",
  
  "triggerConditions": [
    {
      "type": "threshold | absence | change | time_based",
      "description": "When this triggers",

### File 2: `lib/smart-storage.ts` — Adaptive Storage

This module changes what gets saved to MongoDB based on the active mission.

```
How it works:

1. Mission is set with custom storageFields
2. Each perception cycle, INSTEAD of storing the full default schema,
   extract ONLY the fields the mission cares about
3. Apply the change detection filter — only store if something 
   relevant to the mission changed

Key function:

  function buildDocument(perception, mission):
    // Extract only mission-relevant fields from the Gemini response
    // Compare to previous document
    // If nothing mission-relevant changed, return null (don't store)
    // If something changed, return the slim document

Example for "count red backpacks":
  Default storage:  ~2KB per document (full vision + sensors + frame)
  Mission storage:  ~200 bytes per document:
    { timestamp, red_backpack_count: 2, total_backpacks: 5, new_since_last: 1 }

This keeps MongoDB lean and makes Featherless analysis faster
because it's reading clean, mission-focused data.
```

Storage rules by event type:
```
always_store:
  - Mission trigger conditions met (critical events)
  - Mission-relevant count changes
  - First detection after SENTINEL starts
  - User-requested snapshots

store_with_frame:
  - Critical triggers (fall detected, breach, etc.)
  - User explicitly asks "show me what you see"

skip:
  - No change in mission-relevant data since last store
  - Duplicate readings within dedup window (5 seconds default)
```

**MongoDB collection structure:**

Each mission creates its own collection OR uses a `mission_id` field in the shared `detections` collection. Using a mission_id field is simpler:

```javascript
// Document in 'detections' collection during "count red backpacks" mission
{
  timestamp: ISODate,
  mission_id: "backpack_counter_001",
  mission_name: "Red backpack counter",
  
  // Mission-specific extracted data (varies by mission)
  data: {
    red_backpack_count: 2,
    total_backpacks: 5,
    new_red_since_last: 1,
    locations: [{ x: 0.3, y: 0.5 }, { x: 0.7, y: 0.4 }],
  },
  
  // Minimal context (always included)
  context: {
    people_count: 8,
    activity_level: "moderate",
  },
  
  // Only included if trigger fired
  trigger: {
    fired: false,
    type: null,
    message: null,
  },
  
  // Frame only included on critical events
  frame: null,  // or base64 string
}
```

### File 3: `lib/alerts.ts` — External Alerts (Text/Email)

For missions like the grandfather monitor, SENTINEL needs to reach the user when they're not looking at the dashboard.

**Option A — Twilio for SMS (simplest):**
```
Sign up at twilio.com, get a free trial number.
Add to .env.local:
  TWILIO_SID=your_sid
  TWILIO_AUTH=your_auth_token  
  TWILIO_PHONE=+1234567890

API call:
  POST https://api.twilio.com/2010-04-01/Accounts/{SID}/Messages.json
  Body: { To: "+1userphone", From: "+1twilionumber", Body: "ALERT: ..." }

To include a picture (MMS):
  Add MediaUrl: "https://your-app.vercel.app/api/sentinel/frame/latest.jpg"
  (You'd need a route that serves the latest frame as a JPEG)
```

**Option B — Email via Resend (also simple):**
```
Sign up at resend.com, get API key.
POST https://api.resend.com/emails
Body: {
  from: "sentinel@yourdomain.com",
  to: "user@email.com",
  subject: "SENTINEL ALERT: Grandfather may need help",
  html: "<p>Alert details...</p><img src='cid:frame'/>"
}
```

**Option C — Skip external alerts for the hackathon:**
Just have SENTINEL speak the alert through ElevenLabs and show it prominently on the dashboard with a flashing red indicator. You can TELL the judges "in production, this would send a text with the photo" without actually implementing Twilio. Save the time for core features.

**Recommendation:** Go with Option C for the hackathon. Mention Twilio in your pitch as a production feature. If you have extra time, Option A takes about 15 minutes to implement.

### File 4: `app/api/sentinel/mission/route.ts` — Mission CRUD

```
POST /api/sentinel/mission
  Body: { instruction: "count red backpacks" }
  
  Steps:
    1. Send instruction to Gemini for parsing (see mission.ts)
    2. Get back structured mission object
    3. Set as active mission in memory
    4. Store mission definition in MongoDB 'missions' collection
    5. Return the parsed mission so the dashboard can show it
    
  Response: {
    mission_id: "...",
    missionName: "Red backpack counter",
    visionPrompt: "...",
    storageFields: [...],
    triggerConditions: [...],
    mode: "mission"
  }

GET /api/sentinel/mission
  Returns current active mission and all past missions.

DELETE /api/sentinel/mission
  Clears the active mission, reverts to default perception.
```

### File 5: `app/api/sentinel/chat/route.ts` — Conversational Interface

This handles ALL user input — questions, commands, mode switches, and mission creation.

```
POST /api/sentinel/chat
  Body: { message: "how many red backpacks so far?" }
  
  Steps:
    1. Get current perception (latest frame + sensors)
    2. Get active mission context
    3. Query MongoDB for recent mission-relevant events
    4. Send everything to Gemini:
       - Current frame
       - Mission context
       - Recent events from MongoDB  
       - User's message
       - Instruction: "Determine if the user is asking a question,
         giving a new mission, switching modes, or just chatting.
         Respond appropriately."
    5. If Gemini says it's a new mission → call /mission to set it up
    6. If it's a question → answer from context + memory
    7. If it's chat → respond conversationally
    8. Send response to ElevenLabs if voice is enabled
    9. Return text + audio to dashboard
```

### File 6: Modify `app/api/sentinel/perception/route.ts`

Add mission awareness to the existing perception route:

```
Current flow:
  Receive perception → store in MongoDB → check default triggers

New flow:
  Receive perception → check if mission is active
    
  If mission active:
    → Rewrite Gemini prompt using mission.visionPrompt
    → Extract only mission-relevant fields from response
    → Pass through smart-storage filter (only store if changed)
    → Check mission trigger conditions
    → If trigger fires:
        → Speak alert via ElevenLabs (if speakBehavior.onTrigger)
        → Send external alert if configured (text/email)
        → Store event with frame snapshot
    → If no trigger:
        → Store slim document if data changed
        → Skip if nothing relevant changed
  
  If no mission (default):
    → Use default Gemini prompt
    → Store normally
    → Check default triggers (perimeter, person close, crowd)
```

### File 7: `app/api/sentinel/analyze/route.ts` — Mission-Aware Pattern Analysis

```
This route calls Featherless every 30 seconds (or at mission-specified interval).

If mission active:
  → Query MongoDB for recent events WITH the mission_id filter
  → Use the mission's custom featherlessPrompt instead of default
  → Featherless analyzes mission-specific patterns
  → Store analysis in 'analyses' collection with mission_id

Example for "count hallway traffic":
  Default Featherless would say: "3 people detected, activity moderate"
  Mission Featherless says: "142 entries and 128 exits in the last hour.
    Peak flow was 2:30-3:00 PM with 45 entries. Net accumulation of 14
    people suggests the space is gradually filling up."
```
